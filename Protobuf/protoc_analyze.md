# Background

部分采用了kotlin+java混合开发，并使用Protobuf作为数据解析手段的App，在经过proguard混淆后非常难以理解，具体表现为业务逻辑大部分被协程状态机接管，代码零散，调用链中断，相当多的情况下，调用堆栈的第一帧为业务函数，其余帧均为invokeSuspend或者胶水代码；数据序列化与反序列化过程几乎黑盒且不透明，大部分代码在Protobuf解析状态机内流转，所有数据存取过程均有对应桩代码封装，十分零散，最终proguard会将所有业务逻辑与零碎的编译器生成代码统一揉合为无意义的短字母代码，极大增加分析成本。
<br>

因此本文档对这类编译器生成的（近似）桩代码统一规范化分析，总结此类业务无关逻辑的代码特征，以期在后续分析中能够快速识别和排除，减少分析成本。

# ProtoBuf

针对protobuf的分析分为两节，第一节Compiler Gen简单阐述protoc编译器的生成过程以及原理，第二节Code Structure根据第一节的结论给出生成代码的大致框架。在逆向工作中，重点参考第二节来对相关代码片段进行恢复，若对原理有疑问，则可以直接参考第一节的protoc分析，两者相互照应。

跳过编译器前端处理词法语义的部分，关注代码生成逻辑，file.cc内的FileGenerator负责为单个proto文件适配数据结构，并按照该结构的标准生成以供解析和序列化protobuf数据的java文件。总之，编译器生成的java文件与proto文件是一比一适配的，不同的数据结构间的解析和序列化方法不通用。下文称该java文件为解析器文件。
FileGenerator根据编译器提供的基础消息初始化了数个生成器成员：

- message\_generators\_
	
- extension\_generators\_
	
- generator\_factory\_
	

根据其名称含义可以大概知晓这些生成器负责适配数据结构，并生成对应的extend拓展，并通过一个factory成员管理这些生成行为。

```C++
{

  for (int i = 0; i < file_->message_type_count(); ++i) {
    message_generators_[i] =
        generator_factory_->NewMessageGenerator(file_->message_type(i));
  }
  for (int i = 0; i < file_->extension_count(); ++i) {
    extension_generators_[i] =
        generator_factory_->NewExtensionGenerator(file_->extension(i));
  }
}
```

在生成实现类框架后，protoc会继续在实现类内根据proto文件声明的字段及其编号生成对应的常量，代码格式为“public static final int $constant\_name$ = $number$;”，其中$constant\_name为转换大写后的声明字段名称拼接字符串“\_FIELD\_NUMBER”，$number为声明该字段时使用的编号。后续与该字段有关的数据读写/序列化的流程，大部分均要使用到该常量。

```C++
void ImmutablePrimitiveFieldLiteGenerator::GenerateMembers(
    io::Printer* printer) const {
// ...
  printer->Print(variables_,
                 "@java.lang.Override\n"
                 "$deprecation$public $type$ ${$get$capitalized_name$$}$() {\n"
                 "  return $name$_;\n"
                 "}\n");
// ...
  printer->Print(variables_,
                 "private void set$capitalized_name$($type$ value) {\n"
                 "$null_check$"
                 "  $set_has_field_bit_message$\n"
                 "  $name$_ = value;\n"
                 "}\n");
// ...
  printer->Print(variables_,
                 "private void clear$capitalized_name$() {\n"
                 "  $clear_has_field_bit_message$\n");
  JavaType type = GetJavaType(descriptor_);
  if (type == JAVATYPE_STRING || type == JAVATYPE_BYTES) {
    // The default value is not a simple literal so we want to avoid executing
    // it multiple times.  Instead, get the default out of the default instance.
    printer->Print(
        variables_,
        "  $name$_ = getDefaultInstance().get$capitalized_name$();\n");
  } else {
    printer->Print(variables_, "  $name$_ = $default$;\n");
  }
  printer->Print(variables_, "}\n");
```

在生成字段常量编号声明后，protoc会继续通过GenerateMembers函数为字段生成对应的读写以及清除接口，其读接口的代码格式大致为“$public $type$ ${$get$capitalized\_name$$}$()”，写接口格式为“private void set$capitalized\_name$($type$ value)”，清理接口为“private void clear$capitalized\_name$()”。虽然这些接口均以private修饰，但后续均会通过生成的Builder类对外暴露。接口内部实现会根据字段类型有所不同，这里仅以ImmutablePrimitiveFieldLiteGenerator::GenerateMembers作为示例。
